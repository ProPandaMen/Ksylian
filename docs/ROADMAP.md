# Roadmap развития Ksylian

## Цель проекта

Создать self-hosted панель управления Minecraft-серверами, ориентированную не только на Vanilla, Paper и Purpur, но и на Forge, NeoForge, Fabric и модовые сборки.

Основное преимущество Ksylian должно быть не просто в запуске серверов, а в удобном и безопасном управлении модами, обновлениями, резервными копиями и несколькими хостами.

---

### Цель

Сделать текущую версию безопасной и пригодной для постоянного использования на домашнем сервере.

### Задачи

#### Безопасность агента

- [ ] Сделать токен агента обязательным
- [ ] Запрещать запуск агента без настроенного токена
- [ ] Закрыть прямой публичный доступ к порту агента
- [ ] Разрешить подключение только backend-сервису
- [ ] Добавить журнал всех действий агента
- [ ] Добавить защиту от path traversal
- [ ] Запретить передачу произвольных shell-команд
- [ ] Ограничить допустимые директории серверов
- [ ] Добавить rate limit для запросов к агенту

#### Права процессов

- [ ] Перестать запускать Minecraft-серверы от root
- [ ] Создать отдельного системного пользователя для Minecraft
- [ ] Настроить права на серверные директории
- [ ] Ограничить доступ Minecraft-процессов к системе
- [ ] Добавить systemd security options
- [ ] Рассмотреть отдельного пользователя для каждого сервера

#### Исправление существующих типов серверов

- [ ] Разделить установку Vanilla, Paper и Purpur
- [ ] Исправить загрузку настоящего Purpur JAR
- [ ] Проверять checksum скачанных файлов

- [ ] Корректно удалять временные файлы после ошибки
- [ ] Сделать создание сервера атомарной операцией

### Результат этапа

Панель безопасно управляет Vanilla, Paper и Purpur-серверами и может использоваться без постоянного ручного контроля.

#### Архитектура загрузчиков

Создать общий интерфейс:

```text
ServerLoader
├── VanillaLoader
├── PaperLoader
├── PurpurLoader
├── ForgeLoader
├── NeoForgeLoader
└── FabricLoader
```

Каждый загрузчик должен уметь:

- получать список версий;
- устанавливать сервер;
- обновлять сервер;
- определять требуемую Java;
- формировать команду запуска;
- определять установленную версию.

Права должны назначаться как глобально, так и для отдельных серверов.

### Результат этапа

Доступ можно выдавать друзьям и администраторам без передачи полного контроля.

---

## Этап 19. Фоновые задачи

### Цель

Вынести долгие операции из HTTP-запросов.

### Задачи

Добавить очередь задач для:

- создания сервера;
- установки Forge;
- загрузки модпака;
- обновления модов;
- бэкапа;
- восстановления;
- переноса сервера;
- анализа сборки;
- тестового запуска.

Необходимо поддержать:

- [ ] Прогресс
- [ ] Отмену
- [ ] Timeout
- [ ] Повтор
- [ ] Журнал шагов
- [ ] Защиту от параллельных конфликтующих задач
- [ ] Восстановление после перезапуска backend

### Результат этапа

Долгие операции становятся надёжными и не блокируют интерфейс.

---

## Этап 20. AI-помощник

### Цель

Добавить помощника для диагностики и управления сервером.

### Возможности

- объяснение причин падения;
- анализ crash-report;
- поиск несовместимых модов;
- предложение исправлений;
- безопасное обновление сборки;
- создание бэкапа;
- перезапуск сервера;
- анализ производительности;
- поиск client-only модов;
- подготовка отчёта о состоянии сервера.

AI должен работать только через ограниченный набор инструментов:

- получить состояние сервера;
- прочитать лог;
- прочитать crash-report;
- создать бэкап;
- проверить обновления;
- запустить тестовый сервер;
- выполнить безопасный откат.

Необходимо исключить доступ AI к произвольному shell.

### Результат этапа

Администратор может управлять сервером и диагностировать проблемы обычными текстовыми запросами.
